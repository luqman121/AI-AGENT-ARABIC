# تدقيق مرجع Replit Mobile Agent

تاريخ التدقيق: 2026-07-19

## 1. النطاق والمصدر

هذا المستند يسجل التدقيق الإلزامي السابق لأي تعديل تنفيذي. المصدر الأعلى أولوية هو:

`references/replit-mobile-agent/`

تم فتح الصور الست وفحصها بصرياً واحدةً واحدة. جميعها لقطات iPhone بقياس `1284x2778` وتعرض واجهة
داكنة mobile-first. لا يوجد مجلد بديل `reference/Replit` في المستودع.

تمت كذلك مراجعة `AGENTS.md` و`GOAL.md` و`CLAUDE.md` و`README.md` و`docs/design-system.md`
و`docs/implementation-plan.md` وبنية التطبيقات والحزم، ثم تشغيل خط الأساس الكامل: التنسيق، lint،
الأنواع، الاختبارات، والبناء؛ وكلها نجحت قبل بدء التعديل.

## 3. النظام المرئي المستنتج

- خلفية شبه سوداء `#1f1f1f` تقريباً مع أسطح أعلى درجة فقط؛ الحدود رفيعة منخفضة التباين.
- الأبيض غير ناصع للنص الأساسي، والرمادي المتوسط للثانوي، والبنفسجي لون الهوية/الحالة النشطة.
- الأزرق الداكن محجوز لفقاعة المستخدم؛ الأخضر للنجاح؛ الأحمر لنوع PDF.
- زوايا الأسطح متوسطة إلى كبيرة، لكن قائمة التنفيذ نفسها ليست مجموعة بطاقات متنافسة.
- touch targets تقارب 44–48pt.
- padding أفقي يقارب 16–24pt؛ فراغ hero مقصود؛ composer يقترب من عرض الشاشة مع هامش صغير ثابت.
- خط sans حديث، hierarchy محدود: عنوان 26–32pt تقريباً، محتوى 16–20pt، metadata 13–15pt.
- لا توجد ظلال بارزة؛ العمق يأتي من فرق السطح والحدود.
- الحركة المستنتجة: Thinking/Working حي، توسع وانهيار للسجل، auto-scroll محسوب، تحول send ↔ stop،
  وظهور artifact بعد حدث الجاهزية. يجب احترام `prefers-reduced-motion`.

## 4. قواعد النقل إلى العربية RTL

- يعكس ترتيب القراءة والمحاذاة والسهم والـ bubbles، لكن لا يعكس الأيقونات ذات الدلالة العالمية
  عشوائياً.
- يبقى التسلسل الزمني من أعلى إلى أسفل، ويظهر الأحدث قرب composer.
- النص العربي يحاذى يميناً، بينما الأكواد والأسماء والملفات والأحجام في `dir=ltr` مع عزل bidi.
- زر الإرسال يتجه بصرياً مع RTL، والرجوع في جهة بداية الصفحة العربية.
- `Plan` يصبح «خطة»، `Economy` يصبح مستوى تنفيذ مفهوم، وThinking/Working حالات عربية صادقة.
- لا تُترجم أسماء الملفات، الامتدادات، provider-neutral tool labels، أو القيم التقنية ترجمةً تفسدها.
- يجب اختبار الالتفاف والقص عند 390x844 و430x932 مع safe-area والكيبورد الافتراضي.

## 5. التفاعلات المطلوبة

1. إنشاء مشروع من composer واحد: نوع النتيجة + نص + مرفقات + صوت + خيار خطة.
2. انتقال مباشر إلى محادثة المشروع مع حفظ الطلب فوراً.
3. حالة thinking خفيفة ثم عنوان مرحلة قابل للطي.
4. بث أحداث حقيقية من PostgreSQL/SSE؛ القائمة المطوية افتراضياً ويمكن توسيعها.
5. أثناء التشغيل يتحول إرسال إلى إيقاف/إلغاء ويبقى المحرر متاحاً.
6. reconnect يعيد replay من آخر `seq` ولا يفقد الشاشة عند refresh/background.
7. success ينشئ artifact card ورسالة ملخصة وتوقيت عمل/checkpoint.
8. artifact يفتح preview آمن، ويمكن تنزيله ومشاركته والعودة لسجله وإعادة المحاولة.
9. bottom nav العام يبقى ثلاثياً؛ التنقل السياقي داخل المشروع يظهر Agent/Preview/Artifacts/Tasks حسب
   المتاح.

## 6. مقارنة خط الأساس وقت التدقيق

> تحديث ما بعد التنفيذ: هذه المقارنة تسجل حالة المستودع وقت التدقيق الأولي، وليست وصفًا للحالة
> الحالية. نُفذت لاحقًا المرفقات الخاصة مع schema ودورة تخزين كاملة والتحقق من النوع/الحجم، وأصبح
> composer الإنشاء والمحادثة يرسل المرفقات فعليًا. أضيفت كذلك بطاقة artifact بإجراءات المعاينة
> والتنزيل، واستعادة queued runs، ومساحة composer ديناميكية، واختبارات offline/PWA والموبايل. يبقى
> التسجيل/التفريغ الصوتي وتعدد أنواع المخرجات والمشاركة وسجل النتائج الكامل خارج النطاق الحالي.

### نقاط قوة موجودة

- RTL وdark mobile-first وsafe-area وbottom navigation ثلاثي.
- composer مشترك auto-growing.
- معاملات idempotency وtenant scoping وقيد run نشط واحد.
- PostgreSQL مصدر الحقيقة، BullMQ عامل منفصل، `run_events` متسلسلة وSSE مع `Last-Event-ID`
  وheartbeat.
- إلغاء تعاوني، تخزين خاص، preview منفصل الأصل وروابط موقعة قصيرة العمر.
- مسار حقيقي واحد لموقع عربي ثابت مع تحقق sandbox وartifact محفوظ.

### الفجوات الحرجة وقت التدقيق (تاريخية)

1. كان composer يعرض attachment والميكروفون بوصفهما disabled؛ نُفذت المرفقات لاحقًا، بينما بقي
   التسجيل الصوتي خارج النطاق.
2. كان إنشاء المشروع منفصلًا وظيفيًا عن محادثة الوكيل؛ أصبح الآن يستخدم composer موحدًا ويحمل
   المرفقات، بينما لا يزال التسجيل وskill/output contract الموسع خارج النطاق.
3. التشغيل يعرض checklist صغيرة داخل بطاقة plan، لا action log غنياً قابلًا للطي مثل المرجع.
4. الأحداث لا تحمل progress محسوباً، مدة/attempt/tool metadata مناسبة للعرض، أو checkpoint model
   عام؛ الواجهة تعتمد status أكثر من سجل عمل.
5. لم يكن يوجد جدول attachment أو object lifecycle؛ نُفذا لاحقًا بتخزين خاص وحدود وتحقق وتنظيف
   orphan objects.
6. `artifacts` مقيدة DB إلى `static_site` فقط (`packages/db/src/schema/artifacts.ts:58`) وartifact
   واحد لكل run (`artifacts_run_unique`).
7. كانت بطاقة النتيجة تتيح المعاينة فقط؛ أصبحت المعاينة والتنزيل متاحين من البطاقة، بينما المشاركة
   وسجل artifacts الكامل وretry من artifact ما زالت خارج النطاق.
8. لا توجد وجهة artifact مستقلة داخل تنقل المشروع ولا tabs واضحة للمحادثة/المعاينة/السجل.
9. وُسعت الاستعادة لاحقًا لتشمل reconciliation للتشغيلات العالقة ورسائل reconnect ومسودات الجلسة؛ لا
   تزال استعادة رفع جزئي للمرفق غير مكتمل خارج النطاق.
10. `GOAL.md` يصرح بأن M3 مكتمل ويقيد النتيجة الحالية بموقع ثابت، بينما طلب التطوير الحالي يوسع
    النطاق إلى وكيل متعدد المخرجات والمرفقات والصوت. يجب تحديث source-of-truth في التسليم، لا
    الادعاء بأن هذه القدرات كانت مكتملة.
11. لا توجد مشاركة عبر Web Share API أو fallback نسخ الرابط، ولا history/retry UI متكامل.
12. لا توجد حالات permissions للصوت، إلغاء التسجيل، waveform/duration، upload progress، MIME/size
    rejection، أو إزالة المرفق قبل الإرسال.

## 7. قرارات التنفيذ الناتجة

- لا نسخ بصري حرفي لعلامة Replit؛ ننقل أنماط التدفق والكثافة والتفاعل إلى هوية وكيل العربية.
- composer الموحد هو مركز المنتج في create/conversation/running/succeeded.
- progress يجب اشتقاقه من خطوات durable لا مؤقتات تجميلية.
- كل إدخال وartifact وسجل تشغيل يبقى tenant-scoped، خاصاً، ومؤقت الروابط.
- الصوت يحوّل إلى نص كمرفق/رسالة مدققة مع حالة permission واضحة؛ لا تسجيل صامت ولا إرسال تلقائي.
- المرفقات تحتاج schema ومفاتيح تخزين immutable وقيود MIME/size وحالة upload durable.
- سجل النتائج يعرض كل artifacts الناجحة مع preview/download/share/retry، مع تمييز الأحدث.
- أي capability جديدة يجب أن تكون صادقة: إذا لم يوجد مولد فعلي لنوع ما فلا يظهر النوع كمتاح.

## 8. بوابة الخروج من التدقيق

اكتمل التدقيق لأن:

- كل ملفات المرجع الستة فُتحت وسُجلت.
- البنية المرئية والتفاعلية، الحالات، المسافات، RTL، والموبايل وثقت.
- الكود والوثائق وقاعدة البيانات والطابور والتخزين وSSE قورنت بالمرجع.
- خط الأساس الكامل نجح قبل التعديل.

المرحلة التالية المسموح بها: `foundation`، ثم `ux`، ثم `processing`، ثم `outputs` وفق الترتيب
المعتمد.
